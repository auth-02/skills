// 01  — Architecture Overview 3 Layers · 2 Services

Three layers. One source of truth per concern.

Each layer has a distinct audience and query pattern. Never duplicate data across layers — Langfuse owns LLM I/O; stdout owns structure; the canonical line owns the run summary.

LAYER 01

Langfuse — LLM traces

Full LLM I/O tree: prompts, completions, token costs, prompt template versions. System of record for model inputs and outputs. Not duplicated to stdout.

langfuse.tifinrm.com
LAYER 02

Structured events

Narrow JSON stream to stdout marking state transitions, node entry/exit, tool invocations, fin checkpoints, and errors. Each event carries run_id for cross-service correlation.

stdout → ECS logs
LAYER 03

Canonical run line

One fat JSON per run at completion. Aggregates llm_calls, token counts, tools_used, nodes_visited, duration. The grep target for triage — find any run by kind=canonical.

stdout → ECS logs
Data flow: LangGraph node executes → @logged_node emits node.enter/exit → FinClient.call() emits tool.call/result → fin emits checkpoints → router finally emits canonical line. Langfuse trace captured once at run entry via start_as_current_observation.
// 02  — Correlation Model 5 ID Fields

Every event carries the same envelope.

These five fields are the spine of every log line. To reconstruct a full run: filter by run_id and sort by ts.

thread_id
Conversation session. One thread contains multiple requests (e.g. all turns of a chat). Maps to Langfuse session_id. UUIDs only — never the user's name
request_id
One HTTP call into cortex. A thread is a collection of request_ids. Set by the HTTP middleware from x-request-id header (or generated). Propagated to fin as x-request-id
run_id
One LangGraph graph invocation. Fresh UUID per execution — unique across retries, resumes, and parallel threads. The primary grep key for a single agent run. Propagated to fin as x-run-id — all fin events carry it
branch_id
Send() fan-out child. P4 — deferred
Only present when a child branch differs from the root run. On root branches branch_id == run_id — omitted from events to reduce noise. Will appear when Send() fan-out ships
langfuse_trace_id
Drill-down key to Langfuse UI. Set once at run entry. Every event and the canonical line carry it. Click it → see full LLM I/O tree in Langfuse. Absent when Langfuse is disabled or trace creation fails
// 03  — Event Catalog 7 Event Types

Every event is a single JSON line.

All events share the correlation envelope (ts, run_id, thread_id, request_id, langfuse_trace_id). The fields below are the event-specific additions.

node.enter 001

Emitted by @logged_node at the start of every LangGraph node, including codeact_agent and codeact_tools iterations.

{ "event": "node.enter", "ts": "2026-06-01T13:17:45.593Z", "graph": "client_diagnostics", "node": "codeact_agent_client_diagnostics", "phase": "codeact_agent", "run_id": "d871ce07-..." }
node.exit 002

Emitted on successful return. status=interrupted for GraphInterrupt (normal LangGraph control flow — not an error).

{ "event": "node.exit", "graph": "client_diagnostics", "node": "agent_gate_node", "phase": "agent_gate", "status": "ok", // or "interrupted" "duration_ms": 847 }
tool.call 003

Emitted by FinClient.call() in cortex before the HTTP request. metadata.body carries scope + filters — the exact ask sent to fin.

{ "event": "tool.call", "tool": "get_portfolio_performance", "service": "fin", "metadata": { "body": { "scope": {"type":"client","ids":["uuid"]}, "filters": {"timeframes":["SI"]} } } }
tool.result 004

Emitted by FinClient.call() on successful HTTP response, and by fin's tool_span() context manager. Both sides emit — two log lines per successful tool call.

{ "event": "tool.result", "tool": "get_portfolio_performance", "service": "fin", "status": "ok", "duration_ms": 4670, "run_id": "d871ce07-..." }
checkpoint 005

Named mid-execution milestone inside fin service methods. Answers "where inside fin did it fail?" without log-spamming every sub-operation.

{ "event": "checkpoint", "step": "fetching_market_value", "metadata": { "schema": "public_jan31_demo", "path": "client", "timeframes": ["SI"] }, "run_id": "d871ce07-..." }
error 006

Emitted on exceptions in @logged_node, FinClient, tool_span, and on codeact iteration cap. error_msg is first line only — PII and stack frames redacted.

{ "event": "error", "error_type": "IterationCapReached", "error_msg": "codeact stopped after 6 LLM iterations", "graph": "client_diagnostics", "phase": "codeact_agent", "status": "error", "duration_ms": 42180 }
// 04  — Canonical Line Anatomy kind = "canonical"

One line per run. Everything you need for triage.

Emitted in the router's finally block — fires on success, failure, and interrupt. Filter with kind=canonical to see only run summaries.

Identity
run_id langfuse_trace_id graph entrypoint env thread_id request_id user_id org_id
Stable identifiers. env = local/dev/prod. entrypoint = "stream" for all router-initiated runs.
Outcome
status terminal_node failed_phase error_type error_msg
status = "success" | "failed". failed_phase = which phase/node broke (e.g. "iteration_cap", "codeact_agent"). terminal_node = last node that successfully completed.
Timing
started_at ended_at duration_ms
Wall-clock duration of the entire graph execution including all LLM calls, fin round-trips, and python_exec runs.
LLM
llm_calls tokens_in tokens_out models_used
Counted via llm_counting_span — wraps every codeact LLM call and reads response.usage. cost_usd deferred (needs versioned pricing table).
Tools
tool_calls tool_retries tools_used
Incremented by log_event("tool.call") in FinClient — every fin endpoint call is counted. tools_used is the unique set of endpoint names called.
Graph
nodes_visited phases_completed
Counts successful node.exit events — includes each codeact_agent/codeact_tools iteration. phases_completed is a count of distinct phase names that exited ok.
Example canonical line
{ "kind": "canonical", "run_id": "d871ce07-66e4-41a6-b971-f837a6796c39", "graph": "client_diagnostics", "entrypoint": "stream", "env": "local", "langfuse_trace_id": "6dd56bd9ba83eaf606838e4ca17ea942", "status": "success", "terminal_node": "agent_gate_node", "started_at": "2026-06-01T13:19:16.021Z", "duration_ms": 29053, "llm_calls": 4, "tokens_in": 18420, "tokens_out": 1840, "models_used": ["gpt-5.4-mini"], "tool_calls": 2, "tools_used": ["get_portfolio_asset_allocation", "get_portfolio_performance"], "nodes_visited": 11, "phases_completed": 3 }
// 05  — Triage Flow 5 Steps

From something's wrong to here's the fix.

01

Find the run

Every run produces one canonical line. Start there — filter by kind=canonical and narrow by time window, graph, or org_id.

$grep '"kind":"canonical"' cortex.log | grep '"status":"failed"' | tail -20 $grep '"run_id":"d871ce07"' cortex.log | python3 -m json.tool
02

Read the canonical line

The canonical line tells you what happened at a glance. Three fields answer most triage questions immediately:

status — did it succeed?   failed_phase — which phase broke?   duration_ms — was it slow?

If failed_phase=iteration_cap → the agent looped without converging; check the LLM prompt via Langfuse. If tool_calls=0 → the agent never reached fin; look at the codeact agent events.

03

Trace the event trail

Filter all events by run_id and sort by ts. Follow the chain: node.enter → node.exit → tool.call → checkpoint → tool.result. An event without its exit pair is where execution stopped.

$grep '"run_id":"d871ce07"' cortex.log fin.log | sort -t'"' -k8 | python3 -c "import sys,json; [print(json.loads(l)['event'], '-', json.loads(l).get('node') or json.loads(l).get('step') or json.loads(l).get('tool','')) for l in sys.stdin if l.strip()]"
04

Understand the fin failure

If the run called fin, check fin's log for the same run_id. The tool.call shows exactly what cortex asked (scope, filters). The checkpoint sequence shows where inside fin it stopped. The error event shows the exception class and first line.

Example checkpoint trail for a performance failure:

performance_params_resolved → fetching_market_value → market_value_fetched → error: DataError: invalid UUID

The error fires before computing_xirr — so the Redshift fetch succeeded but the org_id was malformed.

05

Deep dive the LLM — Langfuse

For semantic failures (the model got the answer wrong, picked the wrong tool, constructed a bad query) — the structured events won't tell you enough. Use langfuse_trace_id from the canonical line to jump directly to the full prompt/response tree in Langfuse.

→langfuse.tifinrm.com/trace/6dd56bd9ba83eaf606838e4ca17ea942
// 06  — Instrumentation API 4 Primitives

Thin API. Low friction. Just decorate.

@logged_node — New LangGraph node (cortex)
from app.observability.events import logged_node # Works on both async and sync nodes. # graph= defaults to workflow in tracing context # (leave it out on shared nodes used across graphs). @logged_node(graph="ipd_report", phase="my_phase") async def my_new_node(state: IPDReportState, config) -> dict: # node.enter emitted automatically result = await do_work(state) # node.exit emitted with duration_ms return result # on exception: error event emitted, then re-raised
log_checkpoint() — Mid-execution step (fin)
from app.common.events import log_checkpoint async def get_new_tool(self, request: Request) -> Envelope: # Before each meaningful step — keep calls sparse. # Only at steps where a failure would leave you guessing. log_checkpoint("querying_redshift", schema=schema, timeframes=timeframes) rows = await fetch_from_redshift(...) log_checkpoint("redshift_fetched", rows=len(rows)) result = compute_xirr(rows) return build_envelope(result)
logged_span() — Sync sub-operation (cortex)
from app.observability.events import logged_span # Times a block, emits enter + exit events. # Body can contain await — context manager is sync # but that's fine inside an async function. with logged_span("tool.call", "tool.result", tool="parse_ecas") as span: holdings = parse_ecas_pdf(path) # Enrich the exit event: span.add(pdf_pages=holdings.page_count)
llm_counting_span — New codeact engine
from app.observability.events import llm_counting_span @lru_cache(maxsize=1) def get_engine() -> CodeActEngine: return CodeActEngine( workflow_name="my_workflow", # Pass llm_counting_span as the context provider. # It wraps each LLM call in a Langfuse span AND # increments acc.llm_calls + tokens in the accumulator. llm_call_context_provider=llm_counting_span, # ... other args )
// 07  — Redaction Policy PII · Financial · LLM

What stays in logs. What doesn't.

redact_event() runs inside log_event() before JSON serialisation — applied to metadata dicts and error_msg. Policy lives in app/observability/redaction_policy.py.

Data class In stdout events In canonical line In Langfuse
client_name · pan · folio · email · phone sha256:<16-char prefix> — stable for dedup, not reversible sha256 hash Full (access-controlled)
nav_value · market_value · purchase_price · xirr · unrealised_gain [redacted] — financial values dropped Aggregated only (counts) Full (access-controlled)
LLM prompts / completions Not logged Not logged Full — Langfuse is the record
Stack traces Exception class + first line only Exception class only Full
File paths (S3 URIs only) Full — S3 URIs, no local paths Full Full
fin tool args (scope, filters, timeframes) Full in metadata.body — scope/filters visible for debugging tools_used list only Full
Correlation IDs (run_id, request_id, thread_id, user_id, org_id) Full — opaque identifiers by design Full Full
Adding a new field to redact: edit HASH_FIELDS or DROP_FIELDS in cortex/app/observability/redaction_policy.py. Matching is case-insensitive on the field name. Only fields inside the metadata dict of a log event are inspected — top-level identity fields are never redacted.
